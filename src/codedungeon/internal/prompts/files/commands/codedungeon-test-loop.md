# Loldinis Test Loop

Test execution loop for one repo: integration → API → E2E. Failures create fix tasks that re-enter the dev loop via `/codedungeon-loop`.

Deterministic mechanics (branch guard, plan parsing, curl validation, fix-task generation, framework detect) delegated to `codedungeon`. LLM-driven: integration test specialist, playwright test-frontend, test-reviewer.

## Protected-branch rule

```bash
codedungeon git guard --repo "$REPO_DIR"
```

Every commit / push preceded by this. Exit 1 → STOP.

## Non-negotiables

1. **project-startup** (Step 1) — always runs first. Verified: `BASE_URL` captured + project responds on health check.
2. **Test framework detect** — `codedungeon qa detect-framework` resolves run command.
3. **Fix task gen** — on failure, `codedungeon plan append-fix-tasks` writes to PLAN.md + DB.

Skipping any = invalid run.

## Parameters

- `$ARGUMENTS` — task dir (e.g. `.claude/tasks/my-feature/backend/`). Contains PLAN.md + test-task-*.md files.

## Subagent spawn retry

- Wait ≥60s before failing.
- Retry up to 2× (backoff 1s → 2s → 4s).
- Never degrade to inline execution without exhausting retries.

## Prerequisites

- Test tasks generated by Phase 5.6 task-architect (test-task-*.md).
- `codedungeon` bootstrapped.
- Language-specific runtime installed (Python/Node/Go/etc).
- `gh` CLI authenticated.

---

## Architecture

```
MAIN LOOP
  │
  ├─ Step 1: project-startup → bring project UP → $BASE_URL
  │
  ├─ Step 2: Integration tests (language-specialist as test executor)
  │      → failures → code-fix tasks (re-enter /codedungeon-loop)
  │
  ├─ Step 3: API tests (test-api agent → codedungeon qa validate-api per step)
  │      → failures → code-fix OR test-fix OR startup-fix tasks
  │
  ├─ Step 4: E2E tests (test-frontend: Plan → Exec → Review)
  │      → failures → code-fix OR test-fix tasks
  │
  └─ Done → report back to caller
```

## Execution

### Step 0: Validate + parse

```bash
TASK_DIR="$ARGUMENTS"
[ -f "$TASK_DIR/PLAN.md" ] || { echo '{"error":"PLAN.md missing"}'; exit 2; }

META=$(codedungeon plan meta "$TASK_DIR/PLAN.md")
FEATURE_NAME=$(echo "$META" | jq -r .feature)
REPO_NAME=$(echo "$META"    | jq -r .repo)
LANG=$(echo "$META"         | jq -r .lang)

REPO_INFO=$(codedungeon repo resolve "$REPO_NAME")
REPO_DIR=$(echo "$REPO_INFO" | jq -r .path)
```

Detect test framework for later use:
```bash
codedungeon qa detect-framework --path "$REPO_DIR" > /tmp/fw.json
FRAMEWORK=$(jq -r .framework /tmp/fw.json)
RUN_CMD=$(jq -r .run_cmd /tmp/fw.json)
```

### Step 1: project-startup

Spawn `project-startup` agent:

```
$(codedungeon prompts get caveman-ultra)
You are project-startup.
Read your full instructions from: .claude/agents/project-startup.md
REPO_DIR = $REPO_DIR

Your job: bring the project up for testing (docker / podman / local dev server / vercel preview / cloudflare worker).
Capture $BASE_URL and report it back.

max_thinking_tokens: 2000
model: $(codedungeon config model fast)
```

Capture `BASE_URL` from agent output. Export as env.

### Step 2: Integration tests

If `$TASK_DIR` has `test-task-integration-*.md`:
- Spawn language reviewer / specialist in test mode.
- Run `$RUN_CMD` in `$REPO_DIR`.
- Capture pass/fail. Failures → fix task.

### Step 3: API tests

Parse `test-task-api-*.md` into step specs. For each step:

```bash
codedungeon qa validate-api \
  --spec "$TASK_DIR/test-task-api-$N.json" \
  --base-url "$BASE_URL" \
  --token-env TEST_TOKEN > /tmp/result.json

VERDICT=$(jq -r .verdict /tmp/result.json)
```

On `FAIL`:
- Categorize failed checks (status / body_shape / error_quality / etc).
- Write mini review.json with finding → call `codedungeon plan append-fix-tasks`.
- Or spawn `test-api` agent to build richer fix task if the failure is non-obvious.

### Step 4: E2E tests (frontend only)

Skip for backend-only repos. For frontends (nextjs/react/vue):

**Plan**: spawn `test-frontend` MODE=plan.
**Exec**: spawn `test-frontend` MODE=exec — writes + runs Playwright specs, uses auth storageState.
**Review**: spawn `test-reviewer` on the generated .spec.ts files.

Playwright skill resolution:
```bash
PLAYWRIGHT_SKILL_PATH=""
for p in \
  "$HOME/.claude/plugins/local/codedungeon/skills/playwright-e2e/SKILL.md" \
  ".claude/skills/playwright-e2e/SKILL.md"; do
  [ -f "$p" ] && PLAYWRIGHT_SKILL_PATH="$p" && break
done
```

Pass `PLAYWRIGHT_SKILL_PATH` to test-frontend in all 3 modes.

Failures → fix tasks (same pattern as Step 3).

### Step 5: Report

```bash
PASSED=$(jq '.results | map(select(.verdict=="PASS")) | length' all-results.json)
FAILED=$(jq '.results | map(select(.verdict=="FAIL")) | length' all-results.json)
```

Emit:
```
TESTS_EXECUTED: <N>
TESTS_PASSED: <M>
TESTS_FAILED: <K>
FIX_TASKS_CREATED: [paths]
```

## Failure mode

If fix tasks created → caller (phase-6) re-invokes `/codedungeon-loop` on same `$TASK_DIR` to fix them, then re-runs `/codedungeon-test-loop`. Max 3 fix cycles before escalating.
