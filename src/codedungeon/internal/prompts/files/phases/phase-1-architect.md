### PHASE 1: Architect Planning (Isolated Agent)

**Goal**: Spawn the `architect-planner` agent to analyze the codebase and produce a high-level architecture plan.

**Single-repo adaptation**: In single-repo mode, the architect still runs but the plan will have a single `## repo:.` section. The `## cross-repo` section will be empty or omitted. The `## Repos Affected` will list just `.`.

**Bootstrap adaptation**: In bootstrap mode, the architect designs the project from scratch instead of analyzing existing code.

#### Step 1.1: Spawn Architect Planner

Use the Task tool to spawn a `general-purpose` agent with **model: `opus`**.

**If PROJECT_MODE = BOOTSTRAP**, use this prompt:

```
You are the architect-planner designing a NEW project from scratch.

Read your full instructions from: .claude/agents/architect-planner.md

PROJECT REQUEST:
{FEATURE_PROMPT}

STACK: {stack} (lang: {lang})
PROJECT MODE: BOOTSTRAP (empty project — no existing code)

YOUR JOB:
1. Design the project architecture from the ground up:
   - Research best practices for the chosen stack (use WebSearch if needed)
   - Define the directory structure, key modules, and their responsibilities
   - Define the dependency list (packages/crates/modules to install)
   - Define configuration files needed (manifest, config, CI, linter, etc.)

2. Write a comprehensive architecture plan to .claude/plan/arcplan.md

The plan MUST include:
- ## meta section with: project name, repos: ".", lang: {lang}, stack: {stack}, mode: bootstrap
- ## Project Description (what we're building, from the prompt)
- ## Stack & Dependencies (framework, libraries, versions, why each is needed)
- ## Directory Structure (full tree of directories and files to create)
- ## Module Guide (purpose of each module/package, what it exports)
- ## repo:. section with detailed breakdown:
  - Project scaffold (manifest, config files, entry point)
  - Core modules to create
  - Data models / types
  - API layer (if applicable)
  - Database setup (if applicable)
  - Configuration / environment
- ## Implementation Strategy (what to build first, what depends on what)
- ## Execution Order (ordered list of setup steps)
- ## Conventions (naming, file organization, patterns to follow)

IMPORTANT: Since there is no existing code, you are DEFINING the patterns.
Be opinionated — choose well-established conventions for the stack.
The domain planners and specialists will use this plan to generate tasks
that create the project from zero.
```

**If PROJECT_MODE is SINGLE or MULTI** (existing project), use this prompt:

```
You are the architect-planner for this project.

Read your full instructions from: .claude/agents/architect-planner.md

FEATURE REQUEST:
{FEATURE_PROMPT}

AVAILABLE REPOS (from CLAUDE.md):
{REPO_MAP formatted as table: repo | stack | lang | path}
(In single-repo mode, this will be a single entry with repo=".")

PROJECT MODE: {SINGLE or MULTI}

YOUR JOB:
1. Read and understand the project structure:
   - Read CLAUDE.md (root — project overview, workflow, roles, permissions)
   - For each repo in REPO_MAP, read its CLAUDE.md (in single-repo mode, there's only the root CLAUDE.md)
   - Read {repo_path}/docs/CODEBASE_MAP.md for comprehensive repo context
     (architecture, modules, data flows, conventions, navigation guide).
     This map was generated by cartographer and provides a complete overview
     of the repo structure. Use it instead of doing your own deep codebase scan.

2. Scan the codebase for specifics:
   - Use CODEBASE_MAP.md as your primary reference
   - Search for existing similar features/modules
   - Read relevant existing code to understand current patterns
   - Find reference implementations to follow

3. Write a comprehensive architecture plan to .claude/plan/arcplan.md

The plan MUST include:
- ## meta section with: feature name, repos list (which repos are affected — use "." for single-repo)
- ## Feature Description (original prompt)
- ## Codebase Analysis (what you found in the codebase)
- ## Repos Affected (which repos need changes and why — use "." for single-repo)
- ## Implementation Strategy (high-level approach)
- ## repo:{repo_name} section for EACH affected repo (use ## repo:. for single-repo)
- ## cross-repo section (integration points — omit or mark N/A for single-repo)
- ## Execution Order (for single-repo, just list "." as the only entry)
- ## Risks and Edge Cases

BE PRECISE. Less is more — plan the minimum viable change. Read real files. Reference real patterns. Do not add complexity the feature doesn't need.
This plan will be read by domain planners that need concrete guidance.
```

**Test Auth Injection (Step 1.1 continued):** If `TEST_AUTH_MISSING_REPOS` is not empty, you MUST append the following block to the architect-planner prompt above, right before `YOUR JOB:`:

```
PREREQUISITE — TEST AUTH:
The following repos need a dev-only test-login endpoint created
BEFORE any tests can run: {TEST_AUTH_MISSING_REPOS joined by comma}

For each affected repo, include a `### prerequisite:test-auth` section
under `## repo:{name}` with:
- requirement: Create POST /api/auth/test-login endpoint (dev-only, no password)
- requirement: Add ## Test Auth documentation to repo's CLAUDE.md
- constraint: Must be implemented BEFORE all other dev tasks for this repo
- constraint: Endpoint must return { "token": "...", "user": {...} }
- If frontend repo: also requires Playwright auth setup (auth.setup.ts + storageState)
```

If `TEST_AUTH_MISSING_REPOS` is empty, do NOT include this block.

## Tool discipline

Thinking budget inherited from `PHASE_THINKING[1]` in the orchestrator (`codedungeon-dev-cycle.md`). Model tier via `codedungeon config model <reasoning|fast>` (Sprint 7).
